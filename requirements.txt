Практична робота №7-8
Тема: Додавання JavaScript до готового HTML/CSS-мінісайту
1. Мета роботи
Навчити студентів додавати до вже створеного HTML/CSS-мінісайту реальну клієнтську логіку на JavaScript: працювати з DOM-елементами, обробляти події, керувати станом інтерфейсу, валідувати форми, зберігати дані в браузері та динамічно відображати контент.
2. Очікувані результати навчання
• підключати зовнішні JavaScript-файли до багатосторінкового сайту та ініціалізувати код без inline-обробників;
• знаходити елементи через селектори, змінювати класи, атрибути, текст і видимість блоків;
• обробляти події click, input, change, submit, scroll;
• створювати прикладні UI-механіки: мобільне меню, перемикач теми, модальні вікна, акордеон, фільтри;
• посилювати форму зворотного зв’язку через JavaScript-валідацію, лічильник символів та збереження чернетки;
• збирати значення форми в об’єкт або через FormData і показувати користувачу результат відправлення;
• застосовувати localStorage для збереження налаштувань та введених даних;
• писати код так, щоб його можна було підтримувати: функції, зрозумілі імена, перевірка існування елементів, мінімум дублювання.
3. Вихідні умови та база проєкту
Практична робота є прямим продовженням попередніх двох робіт. Студенти не створюють новий сайт, а розширюють уже наявний мінісайт з трьох сторінок. Бажано використати ту саму структуру контенту, яку студенти вже зробили на HTML та оформили за допомогою CSS.
Рекомендована структура проєкту після додавання JavaScript:
lab03-js/
├─ index.html
├─ pages/
│  ├─ about.html
│  └─ contact.html
├─ assets/
│  ├─ img/
│  └─ media/
├─ styles/
│  └─ style.css
├─ js/
│  ├─ main.js
│  ├─ ui.js           (опційно)
│  └─ contact.js      (опційно)
├─ data/
│  └─ services.json   (для додаткових/просунутих завдань)
└─ README.md
	

4. Загальні технічні вимоги
• JavaScript підключати зовнішнім файлом через <script src="..." defer></script>.
• Не використовувати inline-обробники подій на кшталт onclick="..." або oninput="...".
• Усі взаємодії з елементами реалізовувати через addEventListener.
• Якщо якийсь блок є не на кожній сторінці, код має перевіряти його наявність перед роботою.
• Для показу/приховування станів використовувати CSS-класи або атрибут hidden, а не довгі рядки зі style="...".
• Ключі localStorage мають бути осмисленими, наприклад siteTheme, contactDraft, favoritesList.
• Повідомлення про помилки та успіх потрібно показувати в інтерфейсі сторінки; alert допустимий лише для тимчасового тестування.
• Код має бути структурований у функції: initThemeToggle(), initAccordion(), initContactForm() тощо.
• Коментарі у коді — короткі та змістовні: пояснювати не очевидне, а не дублювати код словами.
Рекомендований шаблон ініціалізації:
// main.js
document.addEventListener('DOMContentLoaded', init);
 
function init() {
  initActiveNav();
  initMenuToggle();
  initThemeToggle();
  initBackToTop();
  initAccordion();
  initFilters();
  initModal();
  initContactForm();
}
	

6. Обов’язкові завдання
Нижче подано перелік завдань. Вони підібрані так, щоб студенти відпрацьовували саме ті сценарії, з якими часто стикаються у реальних фронтенд-проєктах. Усі 10 пунктів є обов’язковими для повного виконання практичної.
1. Підключення JavaScript і стартова структура файлів
• Створіть папку js/ та підключіть основний файл main.js на всіх сторінках через defer.
• Організуйте код у вигляді окремих функцій ініціалізації, а не в одному великому блоці.
• Для елементів, яких може не бути на певній сторінці, додайте перевірку if (element) { ... }.
Практична цінність: У реальних проєктах підтримуваність коду починається зі зрозумілої структури, а не зі «скрипта на 500 рядків».
2. Підсвічування активної сторінки в навігації
• На всіх сторінках визначайте поточний URL або шлях файлу й підсвічуйте відповідний пункт меню.
• Активний елемент має отримувати CSS-клас на кшталт .is-active.
• Не дублюйте логіку вручну на кожній сторінці: одна функція має працювати всюди.
Практична цінність: Це стандартний патерн для багатосторінкових сайтів, лендингів, блогів, адмінок і корпоративних сторінок.
3. Кнопка відкриття/закриття мобільного меню
• Додайте кнопку «Меню» або іконку-бургер для малих екранів.
• По кліку меню має відкриватися/закриватися, а кнопка — змінювати атрибут aria-expanded або клас стану.
• Після вибору пункту меню на мобільному пристрої список можна автоматично закривати.
Практична цінність: Мобільна навігація — один із найтиповіших сценаріїв інтерактивності в реальних сайтах.
4. Перемикач світлої/темної теми
• Додайте кнопку або перемикач теми.
• Після зміни теми на <body> має додаватися відповідний клас, наприклад theme-dark.
• Обраний режим потрібно зберегти в localStorage і відновити після перезавантаження сторінки.
Практична цінність: Збереження користувацьких налаштувань між сесіями — типова вимога для сучасних інтерфейсів.
5. Кнопка «Вгору» та динамічний рік у footer
• Додайте кнопку «Повернутися вгору», яка з’являється лише після прокручування сторінки вниз.
• По кліку сторінка має плавно прокручуватися на початок.
• У footer автоматично підставляйте поточний рік через JavaScript.
Практична цінність: Такі дрібні покращення UX часто присутні навіть на простих маркетингових сайтах.
6. Акордеон або вкладки для структурованого контенту
• На index.html або about.html створіть блок FAQ, «Навички», «Послуги» або «Етапи роботи».
• Реалізуйте або акордеон (розгортання/згортання), або таби (перемикання вкладок).
• Видимим повинен бути лише активний контентний блок.
Практична цінність: Акордеони, FAQ та вкладки часто використовуються для економії місця і кращого сканування сторінки.
7. Фільтрація або пошук контенту
• Реалізуйте фільтр для карток, таблиці або списку. Наприклад: «усі / навчання / проєкти / медіа».
• Альтернатива: поле пошуку, яке в реальному часі відфільтровує рядки таблиці або картки.
• Схований елемент не має залишати зайві проміжки в макеті.
Практична цінність: Фільтрація каталогу, списків курсів, товарів або портфоліо — повсякденна задача фронтенд-розробника.
8. Модальне вікно або lightbox
• Додайте модальне вікно для збільшення зображення, перегляду деталей картки або підтвердження дії.
• Модальне вікно має відкриватися по кліку, закриватися по кнопці «Закрити» і по кліку на overlay.
• Прихований стан потрібно реалізувати коректно: без «стрибань» верстки.
Практична цінність: Модальні вікна широко використовуються для перегляду деталей, підтверджень, форм та галерей.
9. Покращення форми: перевірка даних, повідомлення про помилки, лічильник символів
• На сторінці contact.html додайте JavaScript-перевірку: ім’я не коротше 2 символів, email має бути схожим на реальний, повідомлення — не пусте.
• Під полями або поруч з ними показуйте зрозумілі текстові повідомлення про помилки.
• Для textarea додайте лічильник символів, який показує, скільки введено або скільки залишилось до ліміту.
Практична цінність: Якісна валідація та миттєвий зворотний зв’язок — критично важливі для реальних форм заявок, реєстрацій і підписок.
10. Збереження чернетки форми та обробка submit через FormData
• Під час введення даних у форму зберігайте чернетку в localStorage.
• Після перезавантаження сторінки форма має автоматично відновлювати останні значення.
• Під час submit не виконуйте реальне відправлення: через preventDefault() зберіть дані з FormData та покажіть на сторінці блок-підтвердження з введеними значеннями.
• Після успішної «відправки» очистіть localStorage-чернетку.
Практична цінність: Це дуже близько до реальних сценаріїв форм у вебзастосунках: користувач не втрачає введені дані, а інтерфейс показує контрольований результат.
7. Завдання з зірочкою
Нижче наведені завдання підвищеної складності. Їх можна використовувати як резерв на другу пару, для диференціації за рівнем або як домашнє продовження практичної.
Б1. Рендер карток з локального JSON через fetch()
• Створіть data/services.json або data/projects.json із масивом об’єктів.
• Через fetch() завантажте дані та згенеруйте картки на сторінці.
• Передбачте стани «завантаження», «помилка» і «даних немає».
Б2. Передача стану через URLSearchParams
• Після успішної обробки форми виконайте redirect на success.html або ту ж сторінку з параметром у URL.
• Зчитайте параметри через URLSearchParams і покажіть повідомлення на сторінці.
Б3. Сортування таблиці або списку
• Додайте кнопки або select для сортування таблиці за назвою, датою, ціною чи іншим полем.
• Сортування має реально змінювати порядок елементів у DOM.
Б4. Обране / favorites з localStorage
• Дозвольте користувачу позначати картки як «улюблені».
• Список обраного треба зберігати в localStorage і відновлювати при повторному відкритті сайту.
Б5. Кнопка «Очистити чернетку» та підтвердження дії
• Додайте окрему кнопку для очищення введених даних.
• Перед очищенням покажіть власне підтвердження через модальне вікно, а не через стандартний confirm().
8. Мінімальні технічні підказки для студентів
Цей блок можна залишити у методичних матеріалах як коротку шпаргалку. Він не дає готового розв’язку, але задає правильний напрямок реалізації.
8.1. Робота з класами та станами
const button = document.querySelector('.theme-toggle');
const body = document.body;
 
button?.addEventListener('click', () => {
  body.classList.toggle('theme-dark');
});
	

8.2. Отримання даних з форми
const form = document.querySelector('#contact-form');
 
form?.addEventListener('submit', (event) => {
  event.preventDefault();
 
  const formData = new FormData(form);
  const data = Object.fromEntries(formData.entries());
 
  console.log(data);
});
	

8.3. Збереження чернетки в localStorage
const draftKey = 'contactDraft';
 
function saveDraft(data) {
  localStorage.setItem(draftKey, JSON.stringify(data));
}
 
function readDraft() {
  return JSON.parse(localStorage.getItem(draftKey) || '{}');
}
	

8.4. Фільтрація карток за категорією
const cards = document.querySelectorAll('.service-card');
 
function filterCards(category) {
  cards.forEach((card) => {
    const match = category === 'all' || card.dataset.category === category;
    card.hidden = !match;
  });
}

11. Контрольний список перед здачею
• На всіх сторінках немає помилок у консолі браузера.
• Скрипти підключені коректними відносними шляхами.
• Жодна функція не «падає», якщо потрібний елемент відсутній на сторінці.
• Перемикач теми відновлює вибір після оновлення сторінки.
• Форма не відправляється, якщо дані не валідні.
• Чернетка форми відновлюється після перезавантаження сторінки.
• Кнопки, меню, модальні блоки та фільтри працюють повторно, а не лише один раз.
• Код зрозумілий: осмислені імена змінних, немає великої кількості дублювання.
